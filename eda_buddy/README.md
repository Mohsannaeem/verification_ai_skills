# EDA Buddy — UVM Simulation Manager

> Standalone tool repository: [github.com/Mohsannaeem/eda_buddy](https://github.com/Mohsannaeem/eda_buddy)

EDA Buddy turns YAML configuration files into a fully working Makefile for compiling and running SystemVerilog/UVM testbenches. It supports **Questa**, **VCS**, and **Xcelium** simulators.

You write two small YAML files per component (one for build, one for tests). EDA Buddy generates the Makefile with per-test targets, regression groups, live progress reporting, and optional result push to the RMS dashboard.

## 1. Directory Layout

```
eda_buddy/
├── eda_buddy.py               ← entry point
├── project_structure.yaml     ← lists all components and paths
├── README.md
└── scripts/
    ├── makefile_gen.py        ← Makefile generator
    ├── run_report.py          ← test result reporter
    ├── push_result.py         ← RMS result pusher
    └── logger.py

<your-TB-output>/
└── yamls/
    ├── <comp>_build.yaml      ← compile/elaborate settings
    └── <comp>_run.yaml        ← tests, groups, hooks

run/                           ← generated by EDA Buddy (do not edit)
├── Makefile
├── filelists/
│   └── <comp>.f
└── <comp>/
    ├── build/                 ← compiled artifacts
    ├── work/                  ← questa work library
    └── run/
        └── <group>_<ts>/      ← regression session
            └── <test>/
                └── sim.log
```

## 2. Quick Start

### Step 1 — Install Python dependencies
```bash
pip install pyyaml requests
```

### Step 2 — Create `project_structure.yaml`

Copy the template below into `eda_buddy/project_structure.yaml` and fill in your paths:

```yaml
project_name: MY_PROJECT

paths:
  root: /home/user/my_project/run   # where Makefile and logs are written
  filelists: filelists              # relative to root
  makefile:  .                      # relative to root (. = root itself)

hooks:
  pre:  ""
  post: ""

components:
- name: my_vip
  build_cfg:   /home/user/my_project/tb/yamls/my_vip_build.yaml
  runtime_cfg: /home/user/my_project/tb/yamls/my_vip_run.yaml
```

### Step 3 — Create the build and runtime YAMLs (see sections 4 & 5)

### Step 4 — Generate the Makefile
```bash
cd eda_buddy/
python eda_buddy.py --gen-makefile
```

### Step 5 — Compile and run
```bash
cd run/

# Compile
make questa_build_my_vip

# Run a single test
make questa_run_my_vip_my_first_test

# Run a regression group
make questa_run_my_vip_smoke_test
```

## 3. project_structure.yaml

This is the top-level config that ties everything together.

```yaml
project_name: SMART_SOC_PROJECT   # shown in reports

paths:
  root:      /path/to/run         # ALL generated artifacts go here
  filelists: filelists            # subfolder for .f files (relative to root)
  makefile:  .                    # subfolder for Makefile (. = root itself)

# Global hooks run before/after EVERY build and simulation across all components.
# Leave blank ("") to skip.
hooks:
  pre:  ""                        # e.g. "source /tools/questa/setup.sh"
  post: ""                        # e.g. "python notify.py"

components:
- name: axi_stream_master_vip
  build_cfg:   /path/to/tb/yamls/axi_stream_master_build.yaml
  runtime_cfg: /path/to/tb/yamls/axi_stream_master_run.yaml

- name: axi_stream_slave_vip
  build_cfg:   /path/to/tb/yamls/axi_stream_slave_vip_build.yaml
  runtime_cfg: /path/to/tb/yamls/axi_stream_slave_vip_run.yaml
```

| Field | Description |
|---|---|
| `project_name` | Display name used in final reports |
| `paths.root` | Root output directory — Makefile, filelists, and logs all go here |
| `hooks.pre/post` | Shell command run before/after every build AND every simulation |
| `components[].name` | Short identifier used as Makefile target prefix |
| `components[].build_cfg` | Absolute path to the component's build YAML |
| `components[].runtime_cfg` | Absolute path to the component's runtime YAML |

## 4. Build YAML

Controls how the testbench is compiled and elaborated.

```yaml
# my_vip_build.yaml

project:
  name:       my_vip_tb
  top_module: my_vip_tb_top
  timescale:  1ns/1ps
  uvm_version: "1.2"

paths:
  root_dir: /path/to/my_vip_tb    # TB source root (where .sv files live)

compilation:
  include_dirs:                    # +incdir+ paths (relative to root_dir)
  - env
  - sequences
  - agent
  - top

  defines:                         # +define+ macros
  - MY_VIP_INTERFACE_V1_0
  - UVM_NO_DEPRECATED

  sources:                         # source files (relative to root_dir)
    interfaces:
    - top/my_vip_if.sv
    packages:
    - top/my_vip_pkg.sv
    modules:
    - top/my_vip_tb_top.sv
    dut: []                        # leave empty for VIP-only (no DUT)

# Per-component hooks (run before/after THIS component's build only)
hooks:
  pre:  ""
  post: ""

tool_settings:
  questa:
    compile_flags:
    - -sv
    - -mfcu
    - -timescale
    - 1ns/1ps
    elaborate_flags:
    - vopt +acc ${project.top_module} -o db_opt
    coverage_flags:
    - -coverage
    notes: "Do NOT pass uvm_pkg.sv — use QuestaSim built-in UVM (avoids DPI GCC error)"

  vcs:
    compile_flags:
    - -sverilog
    - -full64
    - -debug_access+all
    - -lca
    - -kdb
    elaborate_flags:
    - -top ${project.top_module}
    coverage_flags:
    - -cm line+cond+fsm+tgl+path
```

> **Questa note**: Never add `uvm_pkg.sv` to sources for QuestaSim. Use the built-in UVM library to avoid DPI/GCC runtime errors.

## 5. Runtime YAML

Controls test execution, regression groups, simulation arguments, and result hooks.

```yaml
# my_vip_run.yaml

runtime:
  common_args:                        # passed to every simulation
  - +UVM_VERBOSITY=UVM_MEDIUM
  - +UVM_MAX_QUIT_COUNT=10
  - +UVM_TIMEOUT=100000000           # 100 µs in ps (timescale 1ns/1ps)

  tool_args:
    questa:
    - -batch
    - -do 'run -all; quit'
    vcs:
    - -l simv.log

  debug:
    gui_mode:   false
    dump_waves: true
    wave_format: fsdb
    quiet_sim:  false                 # true = suppress sim output in terminal

test_config:
  active_test: my_first_test
  entry_points:
  - name: my_first_test
    description: "Basic sanity check"
    seed: random                      # or a fixed integer e.g. 12345
    user_args: []                     # extra +define or plusargs for this test only

  - name: my_second_test
    description: "Back-pressure test"
    seed: random
    user_args: ["+BP_CYCLES=8"]

  - name: my_error_test
    description: "Negative test — expects UVM_ERROR"
    seed: 99999
    user_args: []

# Per-component hooks (run before/after EACH simulation for this component)
hooks:
  pre:  ""
  post: ""

# Regression groups — subsets of entry_points to run together.
# Each group creates its own timestamped session directory.
# regression_post_hook is called once after the group finishes,
# with {total}, {passed}, {failed} substituted with actual results.
groups:
  smoke_test:
    regression_post_hook: "python /path/to/eda_buddy/scripts/push_result.py --id MY_VIP_SMOKE --total {total} --passed {passed} --failed {failed}"
    tests:
    - my_first_test
    - my_second_test

  regression:
    regression_post_hook: "python /path/to/eda_buddy/scripts/push_result.py --id MY_VIP_FULL_REG --total {total} --passed {passed} --failed {failed}"
    tests:
    - my_first_test
    - my_second_test
    - my_error_test
```

### Key runtime fields

| Field | Description |
|---|---|
| `runtime.common_args` | UVM plusargs passed to every test |
| `runtime.tool_args.<tool>` | Simulator-specific flags (batch mode, log, etc.) |
| `debug.quiet_sim` | `true` = send sim output only to log file, not terminal |
| `debug.dump_waves` | Enable waveform dumping (use `WAVES=1` on make command) |
| `test_config.entry_points` | List of all tests. Each has a name, seed, and optional user args |
| `groups.<name>.tests` | Subset of entry_points to run as a group |
| `groups.<name>.regression_post_hook` | Shell command executed after the group finishes |

### UVM_TIMEOUT value

With `timescale 1ns/1ps`, UVM works in picoseconds internally:

| Value | Actual sim time |
|---|---|
| `100000000` | 100 µs ← recommended default |
| `1000000000` | 1 ms |
| `10000000000` | 10 ms |

## 6. Generated Makefile Targets

After running `python eda_buddy.py --gen-makefile`, the Makefile at `run/Makefile` contains:

### Build targets
```
questa_build_<comp>      # compile + elaborate with Questa
vcs_build_<comp>         # compile + elaborate with VCS
xcelium_build_<comp>     # compile + elaborate with Xcelium
```

### Per-test run targets
```
questa_run_<comp>_<test_name>
vcs_run_<comp>_<test_name>
xcelium_run_<comp>_<test_name>
```

### Regression group targets
```
questa_run_<comp>_<group_name>
vcs_run_<comp>_<group_name>
xcelium_run_<comp>_<group_name>
```

### Utilities
```
report       # print pass/fail summary across all components
clean        # remove build/, work/, run/ directories (keeps filelists and Makefile)
```

## 7. Running Tests

All commands are run from inside the `run/` directory.

```bash
cd run/

# ── Build ─────────────────────────────────────────────────────
make questa_build_my_vip

# ── Run a single test ─────────────────────────────────────────
make questa_run_my_vip_my_first_test

# ── Run with a fixed seed ─────────────────────────────────────
# Edit seed in the run YAML, then regenerate, or pass via user_args

# ── Run with waveform dump ────────────────────────────────────
make questa_run_my_vip_my_first_test WAVES=1

# ── Run in GUI mode ───────────────────────────────────────────
make questa_run_my_vip_my_first_test GUI=1

# ── Clean everything ──────────────────────────────────────────
make clean
```

After each test finishes you see a one-line result:
```
[ PASS ]  my_first_test   seed=606629218   sim=2625 ns   wall=0:00:18   log: .../sim.log
```

## 8. Regression Groups

Groups are subsets of tests defined in the runtime YAML. Running a group:

1. Creates a timestamped session directory: `run/<comp>/run/<group>_20260625_143021/`
2. Runs all tests in the group sequentially, saving each log inside the session dir
3. Prints a running counter after each test showing `Total | Pass | Fail | Pending`
4. Generates a box-table final report and saves it to `<session>/report.txt`
5. Calls `regression_post_hook` with the actual pass/fail numbers

```bash
# Run smoke_test group (7 tests)
make questa_run_my_vip_smoke_test

# Run full regression (all tests)
make questa_run_my_vip_regression
```

**Live counter output (one line per test):**
```
[ PASS ]  my_first_test    seed=606629218   sim=2625 ns   wall=0:00:18   log: .../sim.log
          Total=7 | Pass=1 | Fail=0 | Pending=6

[ PASS ]  my_second_test   seed=52962454    sim=4555 ns   wall=0:00:21   log: .../sim.log
          Total=7 | Pass=2 | Fail=0 | Pending=5
```

**Final box-table report:**
```
+------+----------------------------------------------------+-----------+--------------+----------+------------+
|    # | Test                                               |    Status |         Seed |  Elapsed |    SimTime |
+------+----------------------------------------------------+-----------+--------------+----------+------------+
|    1 | my_first_test                                      |      PASS |    606629218 |  0:00:18 |    2625 ns |
|      | log: run/my_vip/run/smoke_test_20260625.../my_first_test/sim.log                                      |
+------+----------------------------------------------------+-----------+--------------+----------+------------+
```

## 9. Live Progress and Final Report

### Per-test status (during regression)
Each test prints one line immediately after it finishes:
- `[ PASS ]` in green, `[ FAIL ]` in red, `[TIMEOUT]` in yellow
- Failure signature lines shown inline (up to 3)
- Running counter showing `Total | Pass | Fail | Pending`

### Final report (after all tests)
```bash
# View saved report
cat run/my_vip/run/smoke_test_20260625_143021/report.txt

# Regenerate cross-component summary at any time
make report
```

## 10. Quiet Mode

By default, simulator output streams to the terminal. To suppress it (only the per-test status line appears):

```yaml
# In my_vip_run.yaml
runtime:
  debug:
    quiet_sim: true    # ← add this line
```

Then regenerate:
```bash
python eda_buddy.py --gen-makefile
```

With `quiet_sim: true` the terminal shows only:
```
[QUESTA] Running my_first_test
[ PASS ]  my_first_test   seed=606629218   sim=2625 ns   wall=0:00:18   log: .../sim.log
          Total=7 | Pass=1 | Fail=0 | Pending=6
```

All simulator output is still saved to `sim.log`.

## 11. RMS Integration

EDA Buddy can automatically push regression results to an RMS (Result Management System) dashboard.

### Setup
1. Start the RMS server: `cd <path-to-RMS> && python backend/main.py`
2. Open the RMS GUI and create a regression with the ID you want to use
3. Add a `regression_post_hook` to your group in the runtime YAML:

```yaml
groups:
  regression:
    regression_post_hook: "python /path/to/eda_buddy/scripts/push_result.py --id MY_VIP_REGRESSION --total {total} --passed {passed} --failed {failed}"
    tests:
    - my_first_test
    - my_second_test
```

The `{total}`, `{passed}`, and `{failed}` placeholders are automatically substituted with the actual results from that regression session.

### Graceful failure
If the RMS server is not running, the script prints a warning and exits cleanly — the regression flow is **not** interrupted:

```
[RMS] WARNING: Result was NOT pushed.
[RMS] Reason : Connection refused — server is not running
[RMS] Server : http://localhost:8000
[RMS] To track results, make sure the RMS server is running:
[RMS]   cd <path-to-RMS> && python backend/main.py
[RMS] Then create the regression in the GUI before running tests.
```

### push_result.py options

```
python push_result.py --id MY_REGRESSION --total 46 --passed 44 --failed 2

Options:
  --id      Regression ID (must exist in the RMS GUI)
  --total   Total number of tests
  --passed  Number of tests that passed
  --failed  Number of tests that failed
  --url     RMS backend URL  (default: http://localhost:8000)
  --start   Start time ISO 8601  (default: server uses now)
  --end     End time ISO 8601    (default: server uses now)
  --log     Path to regression log file
```

## 12. Using on a New Machine

The generated Makefile is fully portable. All paths are Make variables at the top of the file.

### To use on a different machine:

**Option A — Re-generate (recommended)**
Clone the repo, update `project_structure.yaml` with your machine's paths, then:
```bash
python eda_buddy.py --gen-makefile
```

**Option B — Edit the Makefile directly**
Open `run/Makefile` and change only these lines at the top:
```makefile
## ======================================================
## PATHS — all paths derived from ROOT
## ======================================================
ROOT                    := /new/machine/path/run
MY_VIP_SRC_DIR          := /new/machine/path/tb/my_vip_tb
```

Everything else — build dirs, work dirs, filelists, run dirs — derives from these two variables. The filelists use `${MY_VIP_SRC_DIR}` which is exported as an environment variable so EDA tools expand it at compile time.

## 13. Hooks

Hooks let you run shell commands before or after specific stages. There are four levels:

| Level | YAML location | Runs |
|---|---|---|
| Global pre | `project_structure.yaml → hooks.pre` | Before every build AND every simulation |
| Global post | `project_structure.yaml → hooks.post` | After every build AND every simulation |
| Component pre | `<comp>_run.yaml → hooks.pre` | Before each simulation for this component |
| Component post | `<comp>_run.yaml → hooks.post` | After each simulation for this component |
| Regression post | `<comp>_run.yaml → groups.<name>.regression_post_hook` | Once after the whole group finishes |

**Example: source setup script before every build**
```yaml
# project_structure.yaml
hooks:
  pre: "source /tools/questa_2023/setup.sh"
  post: ""
```

**Example: send Slack notification when regression finishes**
```yaml
# my_vip_run.yaml
groups:
  regression:
    regression_post_hook: >-
      python push_result.py --id MY_VIP --total {total} --passed {passed} --failed {failed} &&
      curl -X POST $SLACK_WEBHOOK -d '{"text":"Regression done: {passed}/{total} passed"}'
    tests:
    - my_first_test
```
